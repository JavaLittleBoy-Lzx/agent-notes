# 大模型知识过时了怎么解决？

> [!abstract] 一句话回答
> 两种方案：**RAG（检索增强生成）** 运行时查资料，**Fine-tuning（微调）** 训练时注入新知识。**优先选 RAG**，知识更新频繁的场景几乎都选 RAG。

---

## 🆚 RAG vs Fine-tuning 对比

| | RAG | Fine-tuning |
|---|---|---|
| **原理** | 运行时从外部知识库检索相关信息 | 在模型权重层面注入新知识 |
| **知识更新** | ✅ 随时更新，无需重训 | ❌ 需要重新训练 |
| **成本** | 低（无训练成本） | 高（GPU训练成本） |
| **延迟** | 额外检索步骤，稍微慢 | 无额外延迟 |
| **适用场景** | 知识更新频繁、需实时数据 | 任务格式固定、需学习特定模式 |
| **缺点** | 依赖检索质量 | 训练成本高，知识更新麻烦 |

---

## 🔍 RAG 工作流程

```
用户问题：「iPhone 15 最新价格是多少？」
        ↓
向量数据库语义检索：找相关内容片段
        ↓
将检索结果 + 用户问题 一起发给 LLM
        ↓
LLM 基于最新资料生成答案
```

---

## 🎯 什么时候选哪个？

| 场景 | 推荐方案 |
|------|----------|
| 新闻/股价/实时数据查询 | ✅ RAG |
| 需要学习特定行业的行话/表达 | ✅ Fine-tuning |
| 知识库经常更新 | ✅ RAG |
| 任务流程固定（如客服话术） | ✅ Fine-tuning |
| 两者结合 | ✅ RAG（给知识）+ Fine-tuning（调风格） |

---

## 🎯 面试考察点

1. **说出两种方案** — RAG vs Fine-tuning
2. **各自优缺点** — 更新频繁选 RAG，成本高但稳定选 Fine-tuning
3. **RAG 流程** — 检索 → 拼接 → LLM 生成
