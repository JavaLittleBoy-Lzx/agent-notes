# Agent 的基本架构由哪些核心组件构成？

> 来源：微信公众号 - 小林coding
> 原文链接：https://mp.weixin.qq.com/s/FoC5Y4VWe2PaGCtdkcDhTw

## 简要回答

Agent 的基本架构有四个核心组件：**LLM、工具、记忆、规划模块**。

- **LLM** 是整个系统的大脑，负责理解任务和做决策。
- **工具** 让 Agent 能跟外部世界交互，搜索、执行代码、调 API 都靠它。
- **记忆** 让 Agent 在任务执行过程中保持状态，不会「失忆」。
- **规划模块** 负责把复杂目标拆解成可执行的步骤。

这四个组合在一起，才让 Agent 具备了自主完成任务的能力。

---

## 详细解析

### 核心组件概览

一个完整的 Agent 系统，由以下四个核心部件组成：

| 组件 | 类比角色 | 核心职责 |
|------|----------|----------|
| **LLM 核心** | 老板 | 所有决策都经过它拍板 |
| **工具系统** | 外包执行团队 | 负责真正干活（搜索、发邮件等） |
| **记忆系统** | 公司档案室 | 信息的存档和调档 |
| **规划模块** | 项目经理 | 把大目标拆解成可执行的任务单 |

---

### 1. LLM 核心

它是整个 Agent 的**大脑**，所有的输入——不管是用户的指令、工具返回的结果还是记忆里调出来的内容——最终都要经过 LLM 来理解和决策。

LLM 负责判断：
- 下一步该做什么？
- 是继续思考、调用某个工具，还是已经可以给出最终答案了？

**没有 LLM，其他三个组件就是一堆零件，没有人来统一指挥。**

---

### 2. 工具系统

这是 Agent 和**外部世界交互的唯一入口**。

LLM 本身是个纯粹的「语言处理器」，它不能上网、不能读文件、不能执行代码，但这些限制都可以通过工具来突破。

**工具可以是：**
- 搜索引擎
- 数据库查询
- 代码执行器
- 发邮件的 API
- 任何你能用函数封装的能力

#### 工具定义格式（OpenAI Function Calling 格式）

```json
[
  {
    "type": "function",
    "function": {
      "name": "search_web",
      "description": "搜索互联网上的信息",
      "parameters": {
        "type": "object",
        "properties": {
          "query": {
            "type": "string",
            "description": "搜索关键词"
          }
        },
        "required": ["query"]
      }
    }
  }
]
```

#### 工具调用流程

```
LLM 决定调用工具 → 返回 JSON 格式的调用指令 →
你的代码执行这个调用 → 把结果反馈给 LLM
```

**核心分工：**
- **模型负责「决定做什么」**
- **程序负责「真正执行」**

---

### 3. 记忆系统

记忆系统分两层，可以类比人的记忆方式来理解：

#### 短期记忆（Short-term Memory）

- 就是当前这轮对话的上下文，装在 context window 里
- Agent 在一次任务执行过程中靠它记住中间状态
- 比如：第一步搜索到了什么、第二步执行结果是什么
- 就像人的「工作记忆」，容量有限，**任务一结束就清空**

#### 长期记忆（Long-term Memory）

- 通常用**向量数据库**来实现
- 把重要信息 embedding 之后存起来
- 下次用的时候做语义检索拿回来
- 就像人的「长期记忆」，容量大、可以跨天保留
- 但需要主动「回忆」才能调出来

---

### 4. 规划模块

它决定了 Agent 能不能应对复杂任务。

**简单任务**：一步就搞定

**复杂任务**（如「帮我写一份竞品分析报告」）：需要拆解成多个步骤：
1. 搜索竞品资料
2. 整理关键数据
3. 对比分析
4. 撰写报告

**规划模块的两种实现方式：**
- **Plan-and-Execute**：先让 LLM 输出一个完整计划，再逐步执行
- **Error-Recovery**：边执行边规划，根据每步结果动态调整

## Agent 运行核心循环

```python
def agent_run(user_goal: str):
    # 第一步：规划模块上场，把目标拆成步骤列表
    plan = llm.plan(user_goal)
    memory = []  # 短期记忆，用来存每一步的中间结果

    for step in plan:
        # 第二步：LLM 核心做决策，这一步该怎么做？
        action = llm.decide(
            step=step,
            history=memory,  # 把短期记忆传进去，让它知道之前做了什么
            long_term=vector_db.search(step)  # 从长期记忆里捞出相关历史
        )

        if action.type == "tool_call":
            # 第三步：工具系统负责真正执行
            result = tools.execute(action.tool_name, action.args)
            memory.append({"step": step, "result": result})  # 执行结果存入短期记忆

        elif action.type == "final_answer":
            return action.content  # LLM 判断任务完成，返回最终答案
```

### 核心节奏

```
规划 → 决策 → 执行 → 结果存入记忆 → 再决策，循环往复
```

| 组件 | 职责 |
|------|------|
| LLM | 做决策的角色 |
| 工具系统 | 执行者 |
| 记忆系统 | 存储中间结果，不会「失忆」 |
| 规划模块 | 把大目标拆成小步骤 |

## 相关面试题详解

以下为原文汇总的面试题及核心知识点解析：

---

### 面试题 1：字节一面 - 大模型是怎么查天气的？它自己去调 API？

**问题核心**：大模型本身不能上网，如何实现「查天气」这种需要调用外部 API 的功能？

**核心知识点**：Function Calling / Tool Use 机制

**答案要点**：

1. **LLM 本身是纯语言处理器**，不能直接上网、发请求、执行代码
2. 通过 **Function Calling（函数调用）** 机制，LLM 可以：
   - 理解用户意图：「查天气」需要调用外部天气 API
   - 识别需要调用的工具：`get_weather()`
   - 生成结构化的调用参数：`{"location": "北京"}`
3. **程序负责真正执行**：你的代码接收 LLM 返回的调用指令，实际调用天气 API
4. 将 API 返回结果反馈给 LLM，LLM 再生成自然语言回答

**关键分工**：
- LLM 负责「决定做什么」→ 返回 `{tool_name, arguments}`
- 程序负责「真正执行」→ 调用 API，返回结果给 LLM

---

### 面试题 2：字节二面 - 大模型知识过时了怎么解决？微调？RAG？

**问题核心**：如何解决大模型知识陈旧、不了解最新信息的问题？

**核心知识点**：RAG vs Fine-tuning 对比

| 方案 | 原理 | 适用场景 | 优势 | 劣势 |
|------|------|----------|------|------|
| **RAG** | 检索增强生成，运行时从外部知识库获取最新信息 | 知识更新频繁、需要实时信息的场景 | 知识可随时更新、成本低、可追溯 | 依赖检索质量、增加延迟 |
| **Fine-tuning** | 在预训练模型基础上，用特定数据微调 | 任务格式固定、需要学习特定模式 | 推理速度快、适合固定流程 | 成本高、知识更新需重新训练 |

**正确答案**：
- **优先推荐 RAG**：知识更新频繁，不需要重新训练模型
- 复杂任务可结合使用：RAG 提供知识 + Fine-tuning 调整输出风格

---

### 面试题 3：腾讯面试官 - MCP 协议是什么？精通 Agent 开发？

**问题核心**：MCP (Model Context Protocol) 是什么？为什么 Agent 开发必须了解？

**核心知识点**：MCP 协议

**答案要点**：

1. **MCP 是 Anthropic 提出的开放协议**，用于标准化 Agent 与外部工具/数据源的连接
2. **解决的问题**：
   - 之前每个工具提供商都有自己的 API 标准，互不兼容
   - MCP 提供统一标准，Agent 只需实现一次，就能连接所有支持 MCP 的工具
3. **MCP 的三个核心组件**：
   - **Host**：发起操作的 AI 应用（如 Claude Desktop）
   - **Client**：与每个工具保持 1:1 连接的客户端
   - **Server**：暴露特定能力的轻量级程序（如 Slack Server、GitHub Server）

**类比理解**：MCP 就像 USB 接口规范——有了统一标准，任何 USB 设备都能插在任何 USB 接口上。

---

### 面试题 4：阿里技术官 - A2A 和 MCP 到底有什么区别？

**问题核心**：A2A (Agent to Agent) 协议和 MCP 协议的关系和区别？

**核心知识点**：A2A vs MCP 定位差异

| 协议 | 全称 | 解决的问题 | 定位 |
|------|------|------------|------|
| **MCP** | Model Context Protocol | Agent 如何调用外部工具 | Agent ↔ 工具/数据源 |
| **A2A** | Agent to Agent Protocol | 多个 Agent 之间如何协作 | Agent ↔ Agent |

**大白话理解**：
- **MCP**：Agent 怎么用工具（类比：人怎么用电脑）
- **A2A**：Agent 之间怎么聊天协作（类比：员工之间怎么沟通）

**两者关系**：互补协议，完整的 Agent 系统通常需要两者配合：
- MCP 处理 Agent 与外部工具的交互
- A2A 处理多 Agent 协作场景

---

### 面试题 5：腾讯面试官 - 大模型怎么学会调用外部工具？

**问题核心**：LLM 本身只训练过文本，如何学会调用外部 API/工具的？

**核心知识点**：Tool Use 的训练方式

**答案要点**：

1. **Function Calling 的训练方式**：
   - 在预训练阶段：模型学习大量包含「工具调用示例」的文本
   - 在微调阶段：专门用「工具调用」类型的数据集进行训练
   - 典型数据集格式：`用户问题 → {tool_name, arguments} → 工具返回 → 最终回答`

2. **让 LLM 学会调用工具的关键**：
   - **工具描述（Tool Description）**：告诉 LLM 有什么工具、参数是什么
   - **示例训练**：让模型看到大量「什么时候该调什么工具」的样例
   - **输出格式约束**：让模型学会输出结构化的调用指令

3. **Function Calling vs Tool Use**：
   - Function Calling 是 OpenAI 提出的概念，是 Tool Use 的一种实现
   - Tool Use 是更广泛的概念，包括各种让 LLM 操作工具的方法
